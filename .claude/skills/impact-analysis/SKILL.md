# impact-analysis

두 가지 **착수 전 분석 절차**를 표준화한다. 프로젝트 중립이며, 프로젝트 고유 값(추가 영향 축·정본 문서)은 [`.claude/project.json`](../../project.json)에서 읽는다.

## 왜 이 스킬이 필요한가

- **착수 후 재작업 방지**: 영향 범위를 모른 채 코딩에 들어가면 데이터·테스트·정본 문서에 미치는 파급이 뒤늦게 드러나 재작업이 발생한다.
- **방향 결정 비용 절감**: 단일 권장안이나 2옵션만 제시하면 사용자가 비교 기준을 매번 직접 요구해야 한다. 3안 이상 + 비교축을 미리 갖추면 결정이 빠르다.

## 언제 사용하나

### 반드시 사용
- 비자명한 코딩 작업 착수 직전 — 기능 구현·구조 변경·버그 수정·데이터/DB 초기화 등 → **F4 영향도 분석**
- 방법·지향·순서를 문의·검토·제안할 때 → **F3 다안 비교**
- 사용자가 "영향도 분석", "다안 비교", "옵션 비교", "어떻게 접근" 등을 명시 요청할 때

### 사용하지 않음
- 오타 수정·주석 한 줄·문서 1줄 패치 같은 자명·국소 변경
- 이미 기획 문서에 영향 범위·접근법이 확정 기재된 작업 (그 문서를 따르면 됨)

## F3 — 다안 비교 제안

방법·지향·순서를 제안할 때는 **3안 이상**을 아래 표준 템플릿으로 제시한다. 단일안 제시나 2옵션 나열은 사용자의 선택지를 좁힌다.

### 표준 템플릿

```markdown
## [주제] 비교

| 안 | 내용 | 장점 | 단점 | 영향도 | 수행비용 |
|----|------|------|------|--------|---------|
| A  | ...  | ...  | ...  | ...    | ...     |
| B  | ...  | ...  | ...  | ...    | ...     |
| C  | ...  | ...  | ...  | ...    | ...     |

**권장: B** — 이유: [근거를 구체적으로. 왜 다른 안보다 나은지, 어떤 트레이드오프를 수용하는지].
```

비교축 정의:
- **영향도**: 이 안이 코드·데이터·테스트·정본 문서·다른 기능에 미치는 파급 범위. 큰 변경일수록 명시.
- **수행비용**: 구현 분량·난이도·검증 부담 (낮음/중간/높음 + 한 줄 근거).
- **추가 비용축 (필요시)**: 안마다 성격이 다른 비용이 갈리면 `수행비용` 열을 분할하거나 별도 열을 추가한다 — (i) 유지보수 비용, (ii) 기능 추가 비용 (이 안 위에 후속 기능을 얹을 때 난이도), (iii) 구조 변경 비용 (향후 구조 변경을 막거나 비싸게 만드는 정도), (iv) 기술부채 (임시방편·재작업 부담). 모든 안의 비용 성격이 동질적이면 `수행비용` 단일 열로 충분 — 불필요하게 열을 늘리지 않는다.

작성 규칙:
- 안은 **실질적으로 구분되는** 선택지여야 한다. "A를 조금 바꾼 것" 수준의 안은 합친다.
- 권장안은 표 다음에 별도로, **근거와 함께** 제시한다. 근거 없는 권장은 금지.
- 분기 결정을 사용자에게 물을 때는 이 비교를 `AskUserQuestion` 툴에 담고, 권장안을 첫 옵션 + (Recommended) 라벨로 배치한다.

**구체 시나리오·예시 병기**: 추상 비교 표만으로는 사용자가 안 간 차이를 체감하기 어렵다. 안이 갈리는 지점을 **구체 시나리오·입력값·코드 line·수치로 풀어** 제시한다. 사용자가 "이해 못 함"을 명시하면 같은 추상 용어를 되풀이하지 말고 구체값으로 재설명한 뒤 재확인한다. 사용자가 직접 설계 안을 제시한 경우 바로 채택하지 말고 **AI 해석을 먼저 보고**(어떻게 이해했는지 + 타안 비교) → 사용자 확정.

## F4 — 착수전 영향도 분석

비자명한 코딩 작업 착수 전, 아래 **범용 5축 + 프로젝트 도메인 축**을 점검하여 영향을 분석·보고한다. 각 축은 "영향 있음 → 구체 내용" 또는 "영향 없음"으로 결론낸다. 점검은 추정이 아니라 **코드·문서 직접 추적**(grep·Read)에 근거한다.

### 범용 5축 체크리스트

**(a) 코드 — 호출부·의존 모듈 + 기존 구현 중복 확인**
- 수정 대상 함수/클래스/모듈의 호출부를 grep으로 전수 확인했는가
- 시그니처·반환형 변경 시 영향받는 caller 목록
- 함수 제거 시 마이그레이션 경로
- **신규 feature면 그 기능 일부가 기존 모듈에 이미 구현돼 있지 않은지 grep → 실제 net-new 부분만 식별**. scope 폭증·중복 구현 방지의 핵심 레버 — 신규 작업의 절반이 기존 구현에 흡수 가능한 경우가 잦다.

**(b) 데이터 — 스키마·마이그레이션·기존 데이터**
- 영속 데이터 모델(DB 스키마·직렬화 포맷·파일 구조) 변경 여부
- 기존 데이터와의 호환성, 마이그레이션·재생성 필요 여부

**(c) 테스트 — 회귀 가드·신규 테스트**
- 영향받는 기존 테스트
- 신규 테스트 필요 범위
- E2E·통합 검증 재실행 필요 여부

**(d) 정본 문서 — Doc Impact**
- `.claude/project.json`의 `docImpactTargets`에 등재된 정본 문서(스펙·용어집·레지스트리 등) 영향 여부
- 영향 있으면 **동일 PR에 동기 갱신 커밋 포함**
- `docImpactTargets`가 비어 있으면 이 축은 생략 가능

**(e) 기술부채 · 우선순위**
- 이 작업이 만드는 후속 재작업·재캘리브 부담
- 우선순위 조정 비용 (다른 계획·마일스톤 순서에 미치는 영향)
- 임시방편(hard-code 등) 도입 시 후속 정리 항목 명시

### 프로젝트 도메인 축 (설정 주입)

`.claude/project.json`의 `domainAxes` 배열에 프로젝트 고유 영향 축을 추가하면, 그 축들을 위 5축 뒤에 이어 점검한다. 비어 있으면 5축만 사용한다.

> 예 (시뮬레이션 엔진 프로젝트): `domainAxes: ["RNG/결정론", "엔진 버전"]` → 난수 도입 시 결정론 태그·버전 증가·baseline 재캡처 필요 여부를 추가 점검. 각 도메인 축의 구체 점검 항목은 프로젝트 규범 문서에 정의한다.

### 보고 양식

```markdown
## 착수전 영향도 분석 — [작업명]

| 축 | 영향 | 내용 |
|----|------|------|
| (a) 코드 | 있음/없음 | 호출부 N곳 / 의존 모듈 ... |
| (b) 데이터 | 있음/없음 | ... |
| (c) 테스트 | 있음/없음 | 영향 테스트 / 신규 테스트 ... |
| (d) 정본 문서 | 있음/없음 | docImpactTargets 대조 결과 ... |
| (e) 기술부채·우선순위 | 있음/없음 | ... |
| [도메인 축들] | 있음/없음 | ... |

**종합**: [핵심 리스크 1~2개 + 권장 진행 방식. 영향이 크면 작업 분리 제안].
```

분석 결과 영향이 크거나 불확실하면, 착수 전에 F3 다안 비교 또는 `AskUserQuestion`으로 사용자에게 진행 방식을 확인한다.

---

## 두 절차의 관계

F3와 F4는 함께 쓰일 수 있다 — 영향도 분석(F4) 결과 접근법이 갈리면, 그 선택지를 F3 다안 비교로 정리해 사용자에게 제시한다. 반대로 다안 비교(F3)의 각 안에 대한 "영향도" 열은 F4 축 분석의 요약이다.
