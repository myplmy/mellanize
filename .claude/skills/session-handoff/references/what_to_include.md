# 섹션별 수집 기준 — 무엇을 쓰고 무엇은 생략하는가

핸드오프 파일 각 섹션에 **무엇을 포함하고 무엇은 생략할지** 결정하는 기준.

핵심 질문: *"이걸 쓰지 않으면 다음 세션이 실수하거나 시간을 낭비하는가?"*
- YES → 기재
- NO → 생략 (CLAUDE.md/memory가 담당)

---

## § 1. 세션 요약

**포함**: 세션 최대 성과 1~3개 (성과 ≠ 활동), 머지된 PR 표 (번호·제목·머지일·1줄 요약).

**생략**: 세션 중 시도했다가 버린 접근, 일반적 Q&A 교환, CLAUDE.md에 이미 있는 프로젝트 설명.

**판단 기준**: "다음 세션 시작 1분 안에 '이 세션에서 무슨 일이 있었구나' 이해되는가?"

---

## § 2. 현재까지 구현된 사항

**포함**: 완료 범위만 (상세 목록은 CLAUDE.md 위임), 이번 세션에 반영된 새 문서 경로 + 1줄 요약, CLAUDE.md에서 이번 세션에 갱신된 섹션 이름만.

**생략**: 각 항목의 상세 구현 (원본 문서가 소스), 커밋 메시지 전문, CLAUDE.md 전체 diff.

**판단 기준**: "이 섹션만 보고 다음 세션이 '리포 상태가 대충 어디까지 와 있구나' 파악 가능한가?"

---

## § 3. 향후 수행 우선순위

**1순위 — 반드시 포함**:
- **상태**: "사전 조사 완료 / 테스트 RED / baseline 확정" 같은 즉시 착수 가능 표시
- **핵심 사실**: 다음 세션이 바로 써먹을 수치·경로·전제 (예: "호출부 K곳", "완료 시점 baseline 재캡처")
- **PR 구조**: N 커밋 분할 + 각 커밋 1줄 요지
- **테스트 전략**: 어느 테스트가 언제 RED/GREEN으로 바뀌는지

**2순위 이하 — 표 형식**: 순위 | 작업 | 의존 3열만.

**보류 항목**: 이름 + 보류 사유 + 재개 조건 링크.

**생략**: "언젠가 하면 좋을 것" 수준의 아이디어, 사용자가 아직 결정 안 한 옵션 나열.

**판단 기준**: "다음 세션이 1순위를 읽고 즉시 branch를 생성하고 첫 커밋에 착수할 수 있는가?"

---

## § 4. 사용자 성향 / 세션 운영 스타일

**포함 (이번 세션 고유만)**: 처음 확인된 성향, 여러 번 반복된 패턴(강화), 이번 세션 특유의 운영 스타일.

**생략 (CLAUDE.md가 담당)**: 이미 규범화된 협업 규칙(PR/staging 규약, 비평 방식, 권장안 요구 등)은 링크로만.

**좋은 예**:
- ✅ "이번 세션에서 미결 {N}건을 세대별(Q → A → B → C) follow-up으로 분할 진행. 매 세대마다 이전 답변을 문서에 inline 반영 후 다음 질의. 일괄 dump 반려 경험."
- ✅ "PR 머지 직후 사용자가 '다음 세션도 이어받게 정리' 명시 요청 → 핸드오프 프로토콜 신설 계기."

**판단 기준**: "CLAUDE.md 읽은 다음 세션이 이 섹션을 읽으면 **추가로 새 것**을 배우는가? 아니면 중복인가?"

---

## § 5. 이번 세션에서 학습·기억한 사항

**포함**: 이번 세션에서 처음 확정된 프로젝트 구조 핵심, 핵심 수치 + 단일 소스 포인터(값 + 경로), 이번 세션에서 걸렸던 기술 스택 유의사항, 세션 도구 팁.

**생략**: 일반 언어·프레임워크 지식(자명), CLAUDE.md가 담당하는 기본 구조.

**좋은 예**:
- ✅ "{식별자} 단일 소스: `{path}` §N"
- ✅ "{수치}: `{수식}` — 단일 소스: `{path}`"

**나쁜 예**:
- ❌ "Python은 들여쓰기 기반"

**판단 기준**: "다음 세션이 이 정보 없이 작업하면 같은 실수를 반복하거나 같은 조사를 다시 하는가?"

---

## § 6. 다음 세션 착수 가이드

**포함 (필수)**: 명령형 단계(1. X 읽기 → … → N. 첫 커밋), 각 단계의 구체 경로, branch 생성 커맨드 완전 구성, 주의사항.

**생략**: 모호한 지시("~를 고려할 것"), 일반 Git 튜토리얼.

**좋은 예**:
```markdown
1. CLAUDE.md 읽기 (특히 '{섹션 이름}')
2. 본 파일 읽기
3. 기획 문서 재독 — `{path}`
4. git checkout -b {slug} {base-branch}
5. Commit 1: `{생성 파일}` 신설
```

**나쁜 예**:
```markdown
1. 관련 문서 읽기
2. 구현 준비
3. PR 생성
```

**판단 기준**: "이 섹션만 보고 다음 세션이 의문 없이 실행 가능한가?"

---

## § 7. 참조 경로 치트시트

**포함**: 이번 세션에서 자주 언급된 절대·상대 경로, 이전 세션 노트 경로(체인 연장용).

**생략**: 프로젝트 내 모든 경로(과도), 임의 문서 경로.

**판단 기준**: "다음 세션이 경로 찾느라 grep/ls를 반복할 필요가 없는가?"

---

## 공통 원칙

1. **이번 세션 고유 정보에 집중** — 영구 사실은 CLAUDE.md/memory가 담당.
2. **다음 세션의 관점에서 쓰기** — "내가 이 파일을 처음 보는 Claude라면 무엇이 필요한가?"
3. **CLAUDE.md 중복 시 링크** — 규범 문구를 복사하지 말고 "CLAUDE.md '섹션 이름' 참조"로 축소.
4. **§6은 가장 중요** — 착수 가이드가 모호하면 핸드오프 실패. 여기만 완성도 높여도 80% 성공.
