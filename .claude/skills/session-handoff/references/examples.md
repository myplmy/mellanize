# 작성 예시

각 섹션을 **어떤 밀도로** 쓰는지 보여주는 플레이스홀더 기반 예시. 실제 프로젝트 값은 `{...}` 자리에 채운다.

---

## 예시 1: § 1. 세션 요약

```markdown
## § 1. 세션 요약

**기간**: {YYYY-MM-DD} ~ {YYYY-MM-DD}

**최대 성과**:
1. {서브시스템 A} 문서 패치 (관련 문서 N건 수정 + M건 신설) PR #{NN} 머지
2. {기능 B} 구현 사전 조사 완료 — 호출부 {K}곳 매핑, {N} 커밋 PR 구조 설계
3. CLAUDE.md {YYYY-MM-DD} 기준 갱신

**머지된 PR**:

| # | 제목 | 머지일 | 내용 |
|-|-|-|-|
| #{NN} | {title} | {YYYY-MM-DD} | {문서 N건 패치, M건 신설} |
```

**왜 이렇게 썼나**: 성과는 명확한 산출물. 머지된 PR 표는 다음 세션이 `git log`로 바로 매칭 가능.

---

## 예시 2: § 3. 1순위 작업

```markdown
### 1순위: {기능 B} (즉시 착수 가능)

**상태**: 사전 조사 완료. PR #{NN} 머지됨. 구현 branch 미생성.

**핵심 사실**:
- 호출부 {K}곳 ({moduleA}:{n}, {moduleB}:{m}, ...)
- {핵심 식별자·수식} — 단일 소스는 {문서 §위치}
- {신규 상수/버전} 첫 도입
- **경계 사례 {J}건**: {요약}

**PR 구조** ({N} 커밋):
1. {상수/설정 도입}
2. {스키마/모델 변경 + 마이그레이션}
3. {핵심 로직 구현}
4. {호출부 기계적 치환}
5. {구 코드 제거 + 신규 가드 테스트}

**테스트 전략**:
- `{test path}` — Step 1~N 동안 RED, 완료 시 baseline 재캡처 후 GREEN

**Branch 명**: `{slug}`
```

**왜 이렇게 썼나**: 다음 세션이 기획 문서를 다시 읽지 않고도 **핵심 사실 + PR 구조 + 테스트 전략**을 바로 파악. 구체 수치("호출부 K곳", "경계 사례 J건")가 실전에서 즉시 활용됨.

## 예시 3: § 4. 사용자 성향 (이번 세션 고유만)

```markdown
## § 4. 사용자 성향 / 세션 운영 스타일

> CLAUDE.md 'User collaboration style' 등 일반 규범은 거기 참조.
> 여기는 **이번 세션에서 확립·강화된 것만**.

### 이번 세션에서 강화된 패턴
- **다단계 follow-up 반복**: 미결 {N}건을 세대별(Q → A → B → C)로 분할 진행. 매 세대마다 이전 답변을 문서에 inline 반영 후 다음 질의. 일괄 dump 요청 반려.
- **권장안 명시 강제**: 옵션 제시에 대해 매번 "권장: X, 이유: …" 요구. 중립 나열 시 즉시 재작성 지시.

### 이번 세션 운영 특이점
- **selected-text 비평 {N}회**: 답변 중 특정 문장을 quote하며 정확 치환 요구. 전면 재작성 금지.
```

**왜 이렇게 썼나**: CLAUDE.md에 이미 있는 일반 규범은 링크로만. **이번 세션에서 드러난 특이점**만 기재하여 중복 제거.

## 예시 4: § 5. 학습

```markdown
## § 5. 이번 세션에서 학습·기억한 사항

### 프로젝트 구조 핵심
- **{식별자} 단일 소스**: {문서 §위치}
- **{정책} 정의**: {문서 §위치}

### 프로젝트 핵심 수치
- **{수치 이름}**: `{값/수식}` — 단일 소스: {문서 §위치}

### 기술 스택 유의사항
- {이번 세션에서 걸렸던 비자명한 함정 — 예: 특정 라우터 등록 순서, 직렬화 옵션}

### 세션 도구 팁
- gh CLI 경로 해결: `command -v gh` 우선, Windows는 전체 경로 fallback
- PR body에 HEREDOC 사용 권장 (multiline)
- `git add .` / `-A` 금지 — 영구 제외 경로 혼입 위험 (`pr-workflow` 스킬)
```

**왜 이렇게 썼나**: 다음 세션이 이 수치를 다시 찾으려면 문서 여러 개를 뒤져야 함. **단일 소스 포인터**와 함께 기재하면 확인 1회로 끝.

## 예시 5: § 6. 착수 가이드

```markdown
## § 6. 다음 세션 착수 가이드

1. **CLAUDE.md 읽기** — 특히 {섹션 1}, {섹션 2}
2. **본 파일 읽기** — `{handoffDir}/session_{YYYYMMDD}_{topic}.md`
3. **기획 문서 재독** — `{path}` 전체 (특히 {핵심 섹션})
4. **코드 진입점 재확인**:
   - `{path}:{NN-MM}` — {역할}
5. **Branch 생성**:
   ```bash
   git checkout {base-branch}
   git pull
   git checkout -b {slug}
   ```
6. **Commit 1 착수**:
   - `{생성/수정 파일}` — {지시}
   - 테스트 RED 확인 (이 시점 정상)

### ⚠️ 주의사항
- PR base·staging 규약: CLAUDE.md / `pr-workflow` 스킬 참조.
- {이번 세션 고유 주의사항 — 예: "baseline은 마지막 서브커밋 완료 시점 1회만 재캡처"}
```

**왜 이렇게 썼나**: 각 단계가 **실행 가능한 커맨드/경로**로 구성. "Commit 1 착수"까지 구체적. 주의사항은 이 세션에서 확정된 **비자명한 결정**만.

## 안티패턴 (쓰지 말 것)

### 안티패턴 1: 모호한 착수 가이드
```markdown
❌ 1. 관련 문서 검토
   2. 구현 착수
   3. PR 생성
```
→ 어느 문서? 어떤 커밋? 구체 경로·커맨드로.

### 안티패턴 2: CLAUDE.md 규범 복사
```markdown
❌ § 4. 사용자 성향
   - PR base 규약 (…)
   - 단일 소스 원칙 준수
```
→ 이미 CLAUDE.md에 있음. "CLAUDE.md 'User collaboration style' 참조"로 축소.

### 안티패턴 3: 성과 ≠ 활동 혼동
```markdown
❌ 최대 성과:
   1. grep으로 호출부 찾음
   2. 사용자와 Q&A 진행
```
→ 이건 활동. 성과는 "호출부 K곳 매핑 완료" 식의 **산출물**.

### 안티패턴 4: 향후 우선순위에 "언젠가"
```markdown
❌ 2순위: 나중에 하면 좋을 리팩터링들
```
→ 구체 계획 문서가 있는 항목만. 막연한 아이디어는 안 씀.

### 안티패턴 5: 경로 생략
```markdown
❌ 3. 기획 문서 읽기
```
→ 어느 문서? 절대·상대 경로 명시.
